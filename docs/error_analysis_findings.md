## 2. Headline Finding

> **The model is great for typical games and terrible for both extremes.**
> It systematically over-predicts tiny games (≤100 copies) and under-predicts blockbusters (>100k copies). This is classic regression-to-the-mean behaviour on a heavy-tailed target — a model trained to minimise mean RMSE under a roughly log-normal target will hedge toward the middle of the distribution.

The "average error of 53k copies" framing is technically correct but misleading. **Median absolute error is 441 copies** — that's what a typical user of this model should expect. The 53k number reflects a few hundred predictions where the model misses by hundreds of thousands of copies on big-selling games.

## 4. Diagnosis: Why These Failure Modes Exist

1. **Heavy-tail target + mean-RMSE optimisation = regression to the mean.** When you train on `log_copies_sold` with squared-error loss, the model is statistically rewarded for predicting near the conditional mean. On a target that spans 8 orders of magnitude, "near the mean" means "in the 1k-10k range" — which is why both the smallest and largest games get pulled toward that band.

2. **Feature pool can't see post-launch signals.** Wishlists, early reviews, streamer pickup, peak CCU — all the signals that *actually* differentiate a viral hit from a similar-looking flop are post-release and excluded by design. We're predicting the unpredictable.

3. **Quality signals are easily faked.** `tag_count = 20` and `language_count = 17` correlate with sales in the median game but are also easy for a shovelware studio to spam. The model can't distinguish "polished international release" from "tag-stuffed asset flip."

4. **AAA is over-fit because it's rare and consistent.** With only 197 AAA games and `publisher_class_ord` being the strongest feature, the model essentially memorises AAA → big seller. This isn't a problem per se, but it means the strong overall R² is partly a function of one tightly-clustered subgroup.

## 5. Improvement Proposals (ranked by effort × impact)

### High impact, easy

**P1 — Switch XGBoost to a skew-aware objective.**
Default `reg:squarederror` assumes Gaussian residuals; copies-sold residuals are heavy-tailed. Try `objective="reg:tweedie"` with `tweedie_variance_power ≈ 1.5–1.7`, or `objective="reg:gamma"`. These directly address the regression-to-mean problem on right-skewed targets. **One-line change.**

```python
XGBoostModel(objective="reg:tweedie", tweedie_variance_power=1.5)
```

**P2 — Add a tag-spam interaction feature.**
Currently `tag_count` is monotonically "more = better." Add `tag_count × description_length` (or `tag_count / (description_length + 1)`). Real polished games have many tags AND a thoughtful description; spam has many tags AND a generic blurb. Should immediately reduce the "Hobbyist with 20 tags" overprediction failure mode visible in §3.3.

**P3 — Replace `developer_freq` with target-encoded developer history.**
`developer_freq` only knows "how many games this developer released." Replace it with "average `log_copies_sold` of this developer's *prior* games" using k-fold target encoding to avoid leakage. Will help the model distinguish prolific-shovelware factories from prolific-quality studios.

### Medium effort, medium-to-high impact

**P4 — Two-stage model: classifier + bucketed regressor.**
Stage 1: classify a game's sales into one of 4-5 quantile buckets. Stage 2: per-bucket regressor predicts within-bucket. Stops the "model collapses to the mean" pattern in §3.1 by giving each bucket its own loss landscape.

**P5 — Quantile ensemble for uncertainty.**
Train three XGBoosts with `objective="reg:quantileerror"` at q=0.1, q=0.5, q=0.9. Use median for point predictions and the q90-q10 spread as an uncertainty estimate. Doesn't eliminate blockbuster underprediction but makes it visible — useful for the writeup and for any downstream decision-making.

**P6 — Discretise price into tiers.**
Currently `Price` is `log1p`-transformed continuous. Add discrete buckets ({free, $0.99-$4.99, $5-$14.99, $15-$29.99, $30+}) and let trees split on them naturally. F2P games and AAA games behave very differently from mid-priced indies, and the current encoding partially blurs that.

### Low impact or hard

**P7 — Concurrent-release competition.**
Compute "# of games released ±2 weeks of this game's launch" as a market-crowdedness signal. Plausible feature; medium effort to compute. Probably modest gains.

**P8 — NLP features from name and description.**
Catchy 1-word names ("ATLYSS", "Muse Dash"), genre-evocative descriptions. Medium-to-high effort (need embeddings or hand-crafted features), likely modest gains for the ratio of work involved.

**P9 — External signals (wishlists, follower counts, pre-launch buzz).**
Would directly attack the "new game prediction is hard" problem in §3.2 and might catch some viral hits early. Requires data we don't have. Outside scope of this project.

---

## What I'd actually try first

P1 (Tweedie objective). It's a one-line change with the highest expected ROI given the diagnosis: heavy-tail skewness is the single biggest source of error and Tweedie loss is the textbook fix. P2 and P3 are cheap follow-ups if P1 helps.
